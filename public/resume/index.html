Problems I solved

open source kubernetes cluster management

2017 Mar - present

The two problems here were

  • Managing the dozens or possibly thousands of variants of the desired state of a cluster or set of clusters.

  • Leading an open source project from nothing to 6500 github stars and over 100 contributors.

To create a kubernetes cluster, one declares the state of all cluster objects in files, then applies those files to the cluster. To edit the cluster one edits these files and reapplies them.

This creates a new problem: how to manage these edits.

kustomize solves this somewhat recursively by declaring the edits themselves as kubernetes objects.

Kustomize is used by Google and other hundreds of other companies to manage their cluster variants.

I founded and led the project.

Problem: How can distributed servers find each other, securely communicate, and delegate authority.

This project was a predecessor to kubernetes and Google fuchsia.

It required building libraries for service construction and discovery, a grpc-like service and protobuffer language, and a wonderful model of delegatable authorization based on public-key cryptography.

I contributed to the libraries and overall project organization, wrote extensive, tested tutorials, and built demos for android, ios and X11.

  • volley, a distributed openGL app for iOS, android, osX, X11 in Go that demonstrates vanadium’s cross platform power (video).

  • moments, an android photo sharing app demonstrating vanadium services, authentication and discovery.

To test the tutorials I wrote a literate programming system to read markdown, extract fenced code blocks, execute them and report errors ([mdrip]). mdrip is used to test kpt, kustomize and other tools.

Work was awarded with various internal mechanisms.

It was an honor to work with a team including Dave Presotto and Ken Thompson, building the first large system based on Pike’s Go.

Investment in open-source Vanadium was halted in favor of fuchsia. I moved to terrabella.

2005 Lincoln Laboratories  / Group 33 Lexington MA

  * Managed team that built two cornerstone apps for the Reagan Test
    Site. The first gathered sensor (radar, optical, etc.) data from
    the network and presented it in ~30 different widgets (e.g.,
    a radar bore-site view of incoming objects).  The second app was a
    simulator.  It allowed someone to set up a missile attack

    simulated sensors emit data to the network to exercise displays

    etc.). The simulator is central to operator training, as human
    operators control the simulated sensors as they would real
    sensors.

  * Set up iterative test and release cycles new to Lincoln.
    New tested versions of the apps are deployed every two weeks to
    the Reagan Test Site in Kwajalein, rather than once or twice a
    year at unpredictable times.

  * Both applications based on the Eclipse Rich Client Platform (ERCP)
    and SWT, capable of running on either Linux or Windows.  Each
    "display" is an ERCP view.  Infected four other Lincoln groups
    with ERCP technology.
